# 图书管理系统 项目复盘

> 一个用 C 语言写的命令行图书管理系统，是我大一学习 C 语言时做的课程项目。
> 这篇文档用来做项目展示和自我复盘，记录我做了什么、遇到过哪些问题、又是怎么一步步把它整理成一个像样的 GitHub 项目的。

## 项目背景

学完 C 语言的结构体、数组、函数和文件读写之后，我想找一个能把这些知识点串起来的小项目练手，于是写了这个图书管理系统。

它运行在终端里，靠菜单选择数字来操作，可以管理图书信息、借书还书，数据会保存到本地文件，下次打开还在。功能不复杂，但基本覆盖了一个"增删改查 + 文件持久化"的小系统该有的东西，对我来说刚好够练。

一开始所有代码都堆在一个 `图书管理系统.c` 文件里。后来我慢慢把它拆分成多个文件、加上 Makefile、写测试、配 CI，把它从"一份作业"整理成了一个有版本、有测试、有文档的 GitHub 项目。

## 主要功能

程序启动后是一个数字菜单，主要功能有：

1. 添加图书（编号、书名、作者、出版社、价格、总数量、是否推荐）
2. 显示全部图书
3. 按编号查询图书
4. 删除图书
5. 修改图书信息
6. 借书（库存减一）
7. 还书（库存加一）
8. 预算购书推荐（输入预算，自动算出在预算内能买的最优方案）
9. 退出系统（退出前自动保存数据）

其中"预算购书推荐"是我自己加的一个稍微有点意思的功能：用递归（DFS）枚举所有买书组合，在不超过预算的前提下，找出花费最多、本数最多的方案。

图书数据保存在 `books.txt` 里，程序启动时自动读取，每次增删改和借还之后也会自动写回文件。

## 技术点

这个项目用到的主要知识点：

- **结构体**：用 `Book` 结构体保存每本书的所有信息，再用结构体数组存所有书。
- **文件读写**：用 `fopen` / `fprintf` / `fscanf` 把图书数据保存到 `books.txt` 并读回来。
- **函数模块化**：把每个功能拆成单独的函数（如 `addBook`、`borrowBook`、`saveBooksToFile`），`main` 只负责菜单分发。
- **输入处理**：自己写了 `inputInt` / `inputDouble` / `inputString` 这几个输入函数，统一处理"读一整行 + 校验"的逻辑，避免用户乱输入导致程序出错。
- **递归 / DFS**：预算推荐功能用递归枚举所有买书组合。
- **多文件编译**：拆分成头文件 + 多个源文件后，用 Makefile 一键编译。

## 开发过程中遇到的问题

- **一开始单文件太乱**：所有函数和全局变量都挤在一个 `.c` 文件里，越写越长，想找某个函数要翻半天。后来才学着用头文件声明 + 多个源文件拆分，把"接口"和"实现"分开。
- **中文文件名带来的小麻烦**：最初源文件叫 `图书管理系统.c`，在终端里手动编译很容易打错字。这也是后来加 Makefile 的原因之一，直接 `make` 就行。
- **用户输入比想象中难处理**：本来以为 `scanf` 就够了，结果用户输入字母、空行、或者输入流提前结束时，程序会出各种问题（见下面的 bug）。后来才意识到"读一整行再自己解析"比直接 `scanf` 稳得多。
- **读文件时太相信文件内容**：原来的读取代码默认 `books.txt` 格式一定正确，一旦文件被改坏就可能崩溃。后来才补上各种校验。

## 修复过的 bug

这几个 bug 都是真实遇到、并通过单独的 PR 修掉的：

1. **修改图书时库存被重置**（PR #6）
   修改图书信息时，代码直接把库存 `stock` 重设成了总量 `total`，导致"已经借出去的数量"凭空消失。
   修复方法：改之前先算出已借出数量 `borrowed = total - stock`，改完总量后用 `stock = 新total - borrowed` 还原，并且不允许新总量小于已借出数量。

2. **输入流结束时死循环**（PR #19）
   当输入流提前结束（比如按 Ctrl+D，或者用管道喂数据但没以"0"结尾），程序会无限循环打印"输入不能为空，请重新输入"。
   修复方法：让读取函数能区分"空行"和"输入结束(EOF)"，遇到 EOF 时友好地退出程序，而不是空转。

3. **books.txt 损坏导致崩溃 / 读入脏数据**（PR #20）
   读取文件时没有校验，文件为空、首行不是数字、声明的数量超大、或内容残缺时，程序可能崩溃或读进一堆乱数据。
   修复方法：先校验图书数量在合理范围内，再逐条校验每条记录；只要发现格式不对就清空已读数据、给出提示，不让脏数据进入程序。

## GitHub 工作流

虽然只是个人项目，但我特意按照规范的协作流程来做，顺便练习 Git：

- 每做一件事（修 bug、加功能、写文档）都先开一个独立分支，分支名能看出在做什么，比如 `fix-input-validation`、`fix-book-file-loading`、`test-add-more-cases`。
- 在分支上提交，提交信息用 `fix:` / `test:` / `docs:` / `refactor:` 这种前缀，一眼能看出这次提交是干嘛的。
- 改完推到 GitHub，开 Pull Request，再合并回 `main`。
- 重要的需求会先建成 Issue，PR 里用 `Closes #编号` 关联，合并后 Issue 自动关闭。

这样 `main` 上的每一次合并都对应一个清楚的改动，回头看历史很清晰。

## 测试和 CI

为了不让每次改动都得手动跑一遍程序，我写了几个简单的测试脚本放在 `tests/` 里：

- `smoke_test.sh`：最基本的冒烟测试，验证项目能编译、能启动、能正常退出。
- `input_test.sh`：测试非法输入，比如输入字母 `abc`、不存在的菜单项 `99`、空行等，确认程序不会崩、会重新提示。
- `file_test.sh`：测试 `books.txt` 各种损坏场景（空文件、首行非法、数量超大、内容残缺），并且测完会把原来的数据文件还原，不破坏真实数据。
- `run_all.sh`：把上面几个脚本一次性跑完。

同时配了 **GitHub Actions**（`.github/workflows/smoke-test.yml`）：每次 push 和 PR 都会自动在云端装好编译环境、跑 `run_all.sh`。这样如果哪次改动把项目弄坏了，PR 页面上会直接显示失败，合并前就能发现。README 里也加了 CI 状态徽章。

## 版本发布记录

用 Git tag 标记了两个版本：

- **v0.1.0**：第一个相对完整的版本。这之前已经完成了多文件拆分、加上 Makefile、写了冒烟测试和 CI、补齐了文档和运行截图。
- **v0.1.1**：在 v0.1.0 基础上集中修了三个健壮性问题——输入 EOF 死循环、`books.txt` 损坏处理，并补充了更多输入和文件相关的测试用例。

简单说，v0.1.0 是"功能和工程框架搭好了"，v0.1.1 是"把容易出问题的地方修结实了"。

## 后续改进方向

还有一些想做但暂时没做的事：

- 给图书总量、价格加上合法性校验（比如不能是负数），这个准备单独开 Issue 来做。
- 支持按书名、作者模糊查询，而不只是按编号精确查。
- 借还书时记录借阅人信息，而不只是简单地加减库存。
- 图书数量目前用固定大小的数组（上限 100 本），以后可以改成动态分配，去掉这个限制。
- 测试目前还是靠脚本对比输出，比较粗糙，以后可以学着写更细的测试。

## 小结

这个项目本身功能不算复杂，但对我来说收获最大的不只是写出了功能，而是第一次完整地体验了"把一份课程作业，整理成一个有结构、有测试、有版本、有文档的项目"的全过程：学会了拆分代码、用 Git 分支和 PR 管理改动、写测试和配 CI、还有怎么认真地修一个 bug 并记录下来。这些经验对以后做更大的项目应该都用得上。
